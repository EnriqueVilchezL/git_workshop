üß™ Git & GitHub Paired Exercise: Python Functions and Version Control

This tutorial will guide you and a partner through collaborating on a Python project using Git and GitHub. You‚Äôll practice creating a repo, cloning it, writing code, committing changes, pushing to GitHub, and resolving a merge conflict.

If you ever need a refresher on commands or the Git workflow, feel free to refer to the Git Commands section at any time.

üìÅ Step 0: Create a GitHub Repository

Partner A should:

  1. Go to https://github.com and log into your account.
  2. Click the + icon in the top right, then choose New repository.
  3. Name the repository: python-functions-practice
  4. Optionally, add a description.
  5. Leave all checkboxes unchecked (no README, no .gitignore, no license).
  6. Click Create repository.
  7. Copy the HTTPS URL shown (you‚Äôll use it to clone).

üë´ Step 1: Invite Your Partner to the Repository

Still Partner A:

  1. Go to the repository page on GitHub.
  2. Click the Settings tab.
  3. Under Access, click Collaborators.
  4. Click Invite a collaborator.
  5. Type your partner‚Äôs GitHub username and send the invite.

Partner B must accept the invitation before continuing.

üíª Step 2: Clone the Repository

Now both partners should run this at the same time:

‚Üí Run in terminal:

cd python-functions-practice

üêç Step 3: Partner A - Create a Python File

Partner A: create a file named math_utils.py, and define a function that calculates the summation from 1 to n. Before showing the right answer, try to do it!

Answer 
def summation(n):
    summation = 0
    for i in range(n + 1):
        summation += i
    return summation

Now that you‚Äôve created your file, let‚Äôs see what Git thinks is happening. Try to run a status with git. Do you remember how to do it?

Answer

‚Üí Run in terminal:

What do you see? Probably, your output looks like this:

This means that the file is untracked. That is, we have made some changed in the file, but it won‚Äôt be included in the next commit unless you git add it.

Let's try to add it! Do a git add, but don't see the answer until you try it:

Answer

‚Üí Run in terminal:

Now, we have placed added the math_utils.py. But still, we need to make a commit. Try to do a commit with the message "Added summation function".

Answer

‚Üí Run in terminal:

Great! Try to check the GitHub repository you just created.

Hmmm... There is nothing there... Are we missing something?

Answer

Correct! We need to do a push to send those changes into the remote repository.

‚Üí Run in terminal:

Try to see your GitHub repository to check that the file is there.

üë• Step 4: Partner B - Add a New Function

Partner B: create a file named math_utils.py, and define a function that calculates the summation from 1 to n. It has to be a different implementation. Before showing the right answer, try to do it!

Answer 
def summation(n):
    return sum(range(1, n + 1))

Good! Now, try to do the same as your partner. Try to push the changes into the remote repository.

Answer

‚Üí Run in terminal:

git add math_utils.py
git commit -m "Added a mean function"
git push

Whoops... Something went wrong... What is it?

Answer

Partner B did a change in a file that was already created, but the remote file has some changes that Partner B does not have on its local machine.

This is known as a conflict. It happens when you do changes locally but you did not pull the changes that other people did in the remote repository.

üö® Step 5: Expect a Merge Conflict

Partner B: are likely seeing something like

! [rejected] main -> main (fetch first) and many other lines.

This means someone else (Partner A) already pushed changes.

‚Üí Run in terminal:

git pull origin main --allow-unrelated-histories --no-rebase
Note about --no-rebase

In this workshop, we use --no-rebase to keep things simple and avoid confusion while we learn. But in real work environments, using rebase is better because it keeps the history cleaner and easier to understand.

Rebase puts your changes on top of the new changes from the remote repository, making the history a straight line. For now, we don‚Äôt use it to keep things easier.

Besides --no-rebase, the command has other parameters:

  • origin: is the name of the remote repository where we get the changes from (it‚Äôs usually called this). When you run git clone https://github.com/user/project.git, Git automatically creates a ‚Äúremote‚Äù called origin that points to that URL.
  • main: is the main branch of the repository where we get the changes from (don‚Äôt worry about branches for now).
  • --allow-unrelated-histories: is used to join two projects that didn‚Äôt have anything to do with each other before, like when you merge two different folders. In this case, we use it because the repo we cloned was empty, so Git sees two different histories (one with commits ‚Äî the remote ‚Äî and one empty ‚Äî the local).

üß† Remember: In real teams, git pull --rebase is commonly used to avoid problems and keep everything tidy. Here we keep it simple so you can learn without stress.

You‚Äôll see a merge conflict in math_utils.py.

üõ†Ô∏è Step 6: Resolve the Merge Conflict

Open math_utils.py. You‚Äôll see something like:

<<<<<<< HEAD
def summation(n):
    return sum(range(1, n + 1))
=======
def summation(n):
    summation = 0
    for i in range(n + 1):
        summation += i
    return summation
>>>>>>> main

On top, you will see some options, such as Accept Current Change | Accept Incoming Change | Accept Both Changes | Compare Changes.

These are some alternatives that you can choose to solve the conflict. They mean:

  • Accept Current Change ‚Üí Keeps your local changes and discards the remote ones.

  • Accept Incoming Change ‚Üí Keeps the changes from the remote repository and discards your local edits.

  • Accept Both Changes ‚Üí Includes both your changes and the remote changes, stacked one after the other. ‚ö†Ô∏è You‚Äôll often need to edit manually afterward to make the result make sense.

  • Compare Changes ‚Üí Shows a side-by-side diff view to help you decide which changes to keep.

Let's try to accept both changes. If the file made something weird, try to change it so it looks like this:

def summation(n):
    return sum(range(1, n + 1))

def summation_2(n):
    summation = 0
    for i in range(n + 1):
        summation += i
    return summation

Then run:

‚Üí Run in terminal: 

git add math_utils.py
git commit -m "Resolve merge conflict and keep both functions"
git push

‚è∞ Step 7: Partner A - Check the Original Function and Add It Back

Alright! Now let‚Äôs take a little trip back in time to see the very first version of our summation function ‚Äî the one you wrote before your partner started making changes. This is a great way to check if your original function is correct and wasn‚Äôt accidentally changed.

What exactly are we doing here?

  • When we use git checkout <commit>, we temporarily switch our project files to look exactly like they did at that specific commit ‚Äî like rewinding a movie to an earlier scene.
  • This doesn‚Äôt delete anything; it just lets us view or copy code from the past without changing the current state.
  • Once we‚Äôre done looking, we‚Äôll go back to the latest version on the main branch using git checkout main.
  • Then, we‚Äôll add the original function back into the current code, but with a new name so it doesn‚Äôt interfere with your partner‚Äôs changes.

Here‚Äôs what we‚Äôll do, step by step:

  1. First, update your local copy with the latest changes so everything is synced. We don't want any more conflicts! Do you remember how to do it?

    Answer

    ‚Üí Run in terminal:

    git pull

  2. Find the very first commit you made (that‚Äôs like the first snapshot of your project):

    ‚Üí Run in terminal:

    git log --oneline

    This will show a list of commits. Scroll down and find the oldest one at the bottom. Copy its short code (it looks like a mix of letters and numbers, for example: a1b2c3d).

  3. Check out that first commit to see what your code looked like back then:

    ‚Üí Run in terminal:

    git checkout a1b2c3d

    This will make us travel in time to your commit!

    (Replace a1b2c3d with the actual code you copied!)

  4. Open math_utils.py and find your original summation function.

    Copy all the code inside this function ‚Äî this is your original, unmodified version.

  5. Now, come back to the present!

    ‚Üí Run in terminal:

    git checkout main

  6. Open math_utils.py again.
    Paste the original function you copied, but rename it to summation_original to keep it separate from the current one, like this:

    def summation_original(n):
    summation = 0
    for i in range(n + 1):
        summation += i
    return summation

  7. Save the file, then add, commit, and push your changes: Try to do it, you are an expert at this!

    Answer

    ‚Üí Run in terminal:

    git add math_utils.py
git commit -m "Add original summation function as summation_original"
git push

üéâ Now you have both versions of the function in your project! This way, you can compare them, test which one works better, or just keep the original safe. Great job going back in time and learning how to manage your project‚Äôs history like a pro!
